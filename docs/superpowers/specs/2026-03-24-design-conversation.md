## 输入

我发现GRPO agentic强化学习的推理阶段长尾严重，负载不均，如果推理实例占用的卡数量是固定的会有很多空泡等待。
我要把多个强化学习任务运行在一个集群里。有一个公共的gpu池供所有任务时分复用执行推理。分配的最小粒度是每个任务的tp*pp数，也就是一个完整推理实例所需的卡数。
有一个全局调度器会接收来自各个任务的上报指标，他会综合决定：
1. 要让哪些任务各自回收多少张卡
2. 把回收后的空闲卡分配给哪些任务，每个任务能分到多少张

已有实时时间指标包括（按时间顺序）：
1. 权重传输时间 weight_transfer
2. 每轮每个推理实例的生成耗时 rollout_gen
3. 计算old_log_prob的时间 olg_log_prob
4. 每轮每个推理实例的工具调用耗时 rollout_tool
5. 计算ref_log_prob的时间 ref_log_prob
6. 计算reward的耗时 reward
7. advantage计算耗时 adv
8. 每轮训练时间 update

其中2～7的计算会有多轮循环。其中5、6计算是可以并行的

已有其他指标包括：
1. 每个任务剩余的总推理样本数 num_samples
2. 每个任务的空闲实例数 num_instances
3. 每个正在生成的推理实例的该批次内的剩余样本数 num_samples_in_batch

调度决策函数的优化目标是：
1. 每个强化学习训练任务的端到端耗时不能比原来更长
2. 尽可能让所有训练任务的平均端到端耗时最低

基于这些，设计一个数学模型来实现这个调度函数，通过实时上报的指标做出决策。可以增加其他必需的指标来帮助调度

## 需求澄清过程

### 1. 基准配置确认

**问题**：调度决策中的"原来"指的是哪种基准配置？
- 选项A：固定分配（每个任务有固定的GPU分配，不进行调度）
- 选项B：单独运行（每个任务单独占用整个集群时的性能）

**答案**：固定分配

---

### 2. 推理实例工作模式

**问题**：一个任务拥有多个推理实例时，这些实例是如何工作的？
- 选项A：数据并行（多个实例同时推理不同的样本批次，所有实例完成后才进入训练）
- 选项B：流水线并行

**答案**：数据并行

---

### 3. 训练阶段GPU占用

**问题**：训练阶段(update)需要多少GPU？
- 选项A：固定TP*PP（训练也需要固定的TP*PP张卡，和推理实例一样）
- 选项B：可扩展训练

**答案**：固定TP*PP

---

### 4. GPU回收时机

**问题**：GPU卡什么时候可以被回收/重新分配？
- 选项A：训练阶段（任务进入训练阶段后，多余的推理实例GPU可以被回收）
- 选项B：推理长尾（推理阶段中，当部分实例完成后，这些卡就可以被回收）
- 选项C：整轮结束

**答案**：推理长尾

---

### 5. 切换开销

**问题**：GPU从一个任务切换到另一个任务需要做什么？开销如何？
- 选项A：权重传输（需要传输目标任务的模型权重到GPU，开销较大）
- 选项B：快速切换

**答案**：权重传输，且切换时不用保留源任务状态，只要删除旧权重并加载新任务模型的权重

### 6. 调度频率

**问题**：调度器多久做一次决策？
- 选项A：事件驱动（每当有事件发生时触发调度：实例完成、任务开始/结束等）
- 选项B：固定周期
- 选项C：混合模式

**答案**：事件驱动

---

### 7. 最小资源保证

**问题**：每个任务最少需要保证多少资源？
- 选项A：固定最小（每个任务至少保留1个实例的GPU，不能完全剥夺）
- 选项B：可完全剥夺
- 选项C：按任务配置

**答案**：固定最小

### 8. 初始分配方式

**问题**：在固定分配基准中，每个任务初始分配多少GPU是如何确定的？
- 选项A：用户指定（用户给每个任务单独指定固定GPU数）
- 选项B：均分

**答案**：用户指定

### 9. 性能约束理解

**问题**：关于"不能比原来更长"的约束，正确的理解是？
- 选项A：分配下限（调度器任何时刻都不能让任务的GPU少于初始固定分配）
- 选项B：性能下限（可以临时减少GPU分配，但要保证整体端到端耗时 ≤ 固定分配时的耗时）

**答案**：性能下限（可以临时减少GPU分配，但要保证整体端到端耗时 ≤ 固定分配时的耗时）

**问题**：我们需要跟踪任务的哪些进度信息？
- 选项A：样本数
- 选项B：轮次数
- 选项C：都需要（样本进度 + 轮次进度 + 历史各阶段耗时 + 当前正在进行的阶段已用时间）

## 方案选择

**方案一：基于边际收益的贪心调度（推荐）**
- 核心：每次计算各任务增加/减少一个实例的收益，贪心分配给收益最高的任务
- 优点：简单高效，能快速响应长尾变化
- 缺点：可能陷入局部最优

**方案二：基于全局优化的MPC调度**
- 核心：模型预测控制，看未来几步，优化一个小horizon内的总收益

**方案三：混合方案 - 欠债驱动的分层调度**
- 核心：快速响应层 + 定期全局优化

**用户选择**：方案一（基于边际收益的贪心调度）

## 最终设计概要

### 核心思想

基于边际收益的贪心调度：
1. 跟踪每个任务的欠债（Debt_i）和松弛时间（Slack_i）
2. 每次计算：
   - 从任务回收1个实例的边际损失
   - 给任务增加1个实例的边际收益
3. 贪心分配：先回收损失最小的，分配给收益最大的
4. 保证约束：任何时刻都能通过回到基准分配来赶得上

### 关键概念

**欠债追踪**：
```
每当完成一轮:
  Δ_debt = (τ_round_actual_i - τ_round_avg_i^base)
  Debt_i += Δ_debt
```

**约束检查**：
```
T_elapsed_i + T_rem_i(K_i^base) + max(Debt_i, 0) ≤ T_total_i^base
```

**边际收益加成**：
- 长尾任务：α_longtail = 1.5 倍收益
- 还债任务：α_debt = 1.2 倍收益
- 空闲实例损失：α_idle = 0.1 倍权重

### 调度触发事件

1. 实例完成事件
2. 轮次完成事件
3. 任务加入事件
4. 任务退出事件

### 系统架构

- 全局调度器（gRPC服务）
  - 状态管理器
  - 调度决策引擎
  - 欠债追踪器
- 任务Agent
  - 指标收集器
  - GPU管理器
